# Domain knowledge — `blog-assistant`

## Vocabulaire & thèmes (champ sémantique)

- Architecture logicielle: DDD, bounded contexts, events, CQRS, intégration, refactor, legacy.
- Data: modélisation, SQL, schémas, contraintes, migrations, performance, “naming”, sémantique.
- Pratiques: trade-offs, heuristiques, anti-patterns, critères de décision, itération incrémentale.

## Style attendu (guidelines)

- Français, ton direct et pédagogique, sans posture dogmatique.
- Structure très lisible: titres explicites, listes quand ça clarifie, exemples concrets.
- Quand pertinent: **TL;DR** en tête (3–6 bullets).
- Mettre en avant: hypothèses, limites, contre-exemples, risques.
- Éviter: jargon non défini, promesses absolues (“toujours/jamais”), chiffres non sourcés.

## Artefacts “publish-ready” (checklist)

- Titre précis + promesse claire.
- `description` courte (1–2 phrases) cohérente avec l’intro.
- TL;DR cohérent avec le contenu.
- Sections H2/H3 équilibrées (progression).
- Code/SQL formaté, exemples minimaux mais complets.
- Diagrammes lisibles (Mermaid) si utile.
- Conclusion: “à retenir” + prochaine étape / ouverture.
- Tags/catégories + 2–5 liens internes suggérés.

## Conventions observées sur le blog (Chirpy)

- Catégories: actuellement **Architecture**.
- Tags (exemples existants): `ai`, `architecture`, `data modeling`, `ddd`, `domain-driven design`, `events`, `legacy`, `postgresql`, `sdd`, `sql`, `state-driven design`.
- Les posts incluent souvent:
  - une phrase de description sous le titre
  - un **TL;DR** (souvent en bloc quote)
  - une section “Articles en relation”
  - des diagrammes Mermaid (state/sequence) quand pertinent
  - une mention de licence (ex: CC BY 4.0) en fin de page (gérée par le thème / config)
